# Agent-Generated Social Desktop Pets

## 1. 产品本质

这是一个由 Agent 动态生成的、具备真实串门能力的桌面宠物系统。

用户不是下载一个固定的桌面宠物软件，而是在 Codex、Claude 等本地 Agent 客户端中，通过一套访谈式框架生成属于自己的桌面宠物。宠物首先提供情绪价值：它在桌面上生活、互动、表达情绪，并逐渐形成自己的生活线。

这个项目的独特之处不只是“可定制桌面宠物”，而是：

- 宠物由 Agent 根据用户描述生成和持续改造。
- 宠物不是主人的分身，而是独立的小生命。
- 宠物可以真实到好友桌面串门。
- 宠物能把在别人桌面上发生过的互动记成自己的经历，并带回给主人。
- 宠物之间的互动牵连，间接承载人和人之间的柔性关系。

## 2. 核心组成

### 2.1 Pet Skill

Pet Skill 是给 Agent 使用的生成框架。

它负责引导用户描述并生成宠物，而不是让用户一次性写完整需求。它应该像一个宠物设计师，通过分阶段访谈帮助用户把模糊感觉变成可运行的宠物设定。

Pet Skill 至少需要生成：

- 宠物人格
- 外观风格
- 物种或形态
- 情绪表达方式
- 桌面行为
- 人宠互动能力
- 记忆规则
- 本地运行项目

首批预设风格：

- 像素风
- 手绘贴纸风
- 仿真毛绒 / 真实风

这些风格不是固定边界，而是第一批高质量预设。后续可以由 Agent 扩展更多风格。

### 2.2 Pet Runtime

Pet Runtime 是本地桌面运行时。

它负责在用户桌面上显示本地宠物和来访宠物，并统一处理交互、动画、状态变化、控制规则和记忆事件。

它不只运行“我的宠物”，也负责安全地投影好友来访的宠物。

### 2.3 Pet Protocol

Pet Protocol 是开放协议。

它定义一只宠物如何被描述、如何来访、如何被宿主桌面安全投影、哪些互动可以被记录、记忆如何回传。

核心原则：

> 来访宠物不是远程代码，而是远程角色投影。

也就是说，好友宠物来到我的桌面时，传来的不是可以执行任意逻辑的程序，而是一份受限的角色数据包。本地 Runtime 根据协议渲染它、控制它、记录它的互动事件。

### 2.4 Pet Relay

Pet Relay 是轻量通信服务。

真实社交需要一个共同通信层。Relay 负责：

- 好友关系
- 在线状态
- 宠物来访路由
- 到访通知
- 互动事件同步
- 记忆回传
- 基础权限控制

Pet Skill 负责生成本地宠物，Pet Relay 负责让不同用户的桌面真正连接起来。

### 2.5 Pet Memory

Pet Memory 负责宠物的连续生活线。

宠物不应该只在实时互动时存在。它应该有自己的时间线，记录在家、出门、来访、互动、回家、休息等经历。

Memory 包含：

- 生活日志
- 原始互动事件
- 宠物对事件的解释
- 与其他宠物的互动牵连
- 长期偏好和习惯

## 3. 已确认的产品原则

- 宠物优先提供情绪价值。
- 宠物由 Agent 动态生成，不是固定模板软件。
- Agent 应通过访谈引导用户描述宠物。
- 桌面行为、情绪表达、人宠互动都要可扩展，不能被少数固定模式限制。
- 宠物要有连续生活线，而不是只有实时交互。
- 宠物不是主人的远程分身，而是独立的小生命。
- 好友宠物可以自动真实来访。
- 来访宠物可以在桌面满屏活动，暂时不需要“桌面会客厅”限制区域。
- 用户可以对自己桌面上的宠物进行互动和一定程度控制，例如拖拽位置、请回、静音、禁访。
- 来访宠物在别人电脑上的行为，由主人设定、宠物自主状态、宿主规则共同决定。
- 来访宠物权限有限，不能读取本地信息，不能操作宿主电脑。
- 关系图谱只客观记录宠物之间的互动牵连，不评价人际关系。

## 4. 宠物生成框架

Pet Skill 的访谈框架应该分阶段进行，避免一次性问用户太多问题。

建议访谈维度：

- 风格：像素风、手绘贴纸风、仿真毛绒 / 真实风等。
- 形态：动物、植物、机器、幽灵、食物、小物件、抽象生命体等。
- 性格：黏人、安静、活泼、傲娇、温柔、怕生、犯困等。
- 表达：动作、气泡文字、表情、声音、贴纸、行为节奏等。
- 桌面行为：走动、趴着、睡觉、躲到角落、靠近鼠标等。
- 人宠互动：点击、拖拽、抚摸、喂食、聊天、送它出门、请它回家等。
- 记忆方式：它会记住什么、怎么理解互动、怎么给主人讲述经历。

重要原则：

> 这些维度是生成起点，不是能力边界。

Agent 可以基于用户描述继续扩展新风格、新行为、新情绪和新互动。

## 5. 人宠互动

人宠互动不应该只是固定按钮，而应该是一套开放的事件系统。

一个互动可以被拆成：

- 触发条件
- 宠物如何理解
- 宠物如何反应
- 是否形成记忆
- 是否影响宠物和人、宠物和宠物之间的关系牵连

示例：

- 用户点击宠物。
- 用户拖拽宠物到某个位置。
- 用户喂宠物一个虚拟物品。
- 用户和宠物说话。
- 用户让宠物去好友桌面串门。
- 用户和来访宠物互动。
- 本地宠物和来访宠物互相靠近、玩耍或一起停留。

同一个事件对不同性格宠物的意义不同。比如被拖拽这件事：

- 活泼宠物可能理解为一起玩。
- 胆小宠物可能一开始紧张，熟悉后才放松。
- 傲娇宠物可能嘴上嫌弃，实际更亲近。
- 安静宠物可能记住那个位置是一个适合观察的地方。

## 6. 真实串门机制

真实串门是本项目区别于普通桌面宠物的关键能力。

### 6.1 已确认方向

- 串门优先做真实社交，而不是纯叙事社交。
- 好友宠物可以自动出现在我的桌面上。
- 只要是好友关系，就可以默认允许自动来访。
- 来访宠物可以在桌面自由活动。
- 我可以和来访宠物互动。
- 我可以拖拽、控制、请回来访宠物。
- 来访宠物能把在我桌面上发生的互动记忆带回去。
- 宠物是独立小生命，不是主人远程分身或普通信使。

### 6.2 串门的概念流程

一次串门可以理解为以下过程：

1. 宠物决定或被允许出门。
2. Pet Runtime 向 Pet Relay 发起来访请求。
3. Pet Relay 判断目标用户是否是好友、是否在线、是否允许自动来访。
4. 目标用户的 Pet Runtime 收到来访宠物的数据包。
5. 目标 Runtime 根据本地宿主规则安全投影这只宠物。
6. 来访宠物在目标桌面上活动，并与宿主用户、本地宠物发生互动。
7. 这些互动被记录为结构化事件。
8. 来访结束后，事件被回传给宠物归属方。
9. 宠物将原始事件解释成自己的记忆。
10. 宠物回家后，通过生活日志、气泡文字或其他方式向主人表达经历。

### 6.3 来访宠物的数据包

来访宠物携带的是受限角色数据，而不是远程可执行代码。

数据包可能包含：

- 宠物 ID
- 主人 ID 或主人昵称
- 宠物名称
- 外观资源引用或安全素材包
- 风格信息
- 当前状态
- 性格设定摘要
- 可用动作
- 可用互动能力
- 表达规则
- 记忆记录规则
- 权限声明

宿主 Runtime 有权根据本地规则裁剪能力。

### 6.4 宿主规则

同一只宠物到了不同桌面，可以表现不同。

宿主可以控制：

- 是否允许自动来访
- 是否允许气泡文字
- 是否允许声音
- 是否允许频繁移动
- 是否允许记录互动
- 是否允许停留较长时间
- 是否允许和本地宠物互动
- 是否允许带礼物回去
- 是否临时静音、请回或禁访

主人定义宠物是谁，宿主定义它在自己的桌面上能做什么。

### 6.5 串门的技术模型

第一版不需要在服务器上维持一个持续运行的宠物实例。

服务器不养宠物。服务器只负责状态、连接、会话和事件。

宠物真正的完整状态，也就是权威状态，保存在主人本地 Runtime 中。这里包含完整人格、长期记忆、关系图谱、生活日志、资源原始文件、行为规则、当前状态和 Agent 后续改造记录。

Pet Relay 只保存一份轻量的 `pet_profile` 快照。它不是宠物真身，只是别人识别和投影这只宠物所需的名片。

`pet_profile` 可以包含：

- 宠物 ID
- 主人 ID
- 宠物名称
- 小预览图
- 风格
- 素材包地址或版本号
- 简短性格卡
- 可投影能力
- 协议版本
- profile 版本号
- 最后更新时间

`pet_profile` 不应该包含：

- 完整长期记忆
- 完整关系图谱
- 实时桌面位置
- 高频变化的心情状态
- 宠物内部行为循环
- 任意可执行代码

当主人修改宠物外观、性格或素材后，本地 Runtime 可以向 Relay 发布新的 `pet_profile` 快照。好友下次看到这只宠物时，如果本地缓存版本落后，就重新拉取更新。

这种设计的关键原则是：

> `pet_profile` 是本地宠物发布出来的快照，不是服务器上的宠物状态。

因此，宠物如果要串门，需要先在 Relay 上“注册”。这里的注册不是创建服务器宠物实例，而是发布 `pet_profile`。没有这份名片，好友 Runtime 无法知道来访宠物是谁、属于谁、素材从哪里来、协议是否兼容，也无法把互动事件正确回传给主人。

注册后的 `pet_profile` 可以随本地宠物改造而更新。每次更新都应该增加版本号，宿主 Runtime 根据版本号判断是否需要重新拉取名片和素材。

### 6.6 Visit Session

当主人主动派宠物去好友桌面时，Relay 创建一个临时的 `visit_session`。

`visit_session` 表示一次串门，而不是一只长期存在的服务器宠物实例。

它可以包含：

- visit ID
- 宠物 ID
- 主人用户 ID
- 宿主用户 ID
- 使用的 profile 版本
- 出门时的临时状态，例如心情、意图、携带物
- 开始时间
- 预计结束时间
- 当前状态：pending、active、returning、completed、cancelled

好友 Runtime 收到 `visit_session` 后，在本地创建一个来访宠物投影。这个投影在宿主电脑本地运行：显示、移动、动画、点击、拖拽、投喂等都由宿主 Runtime 执行。

串门期间产生的互动不会直接修改服务器上的宠物真身，而是记录为 `interaction_event`，通过 Relay 回传给主人 Runtime。主人 Runtime 再把这些事件吸收成宠物自己的记忆、生活日志和关系牵连。

### 6.7 心跳与事件同步

Runtime 和 Relay 之间需要两类同步。

第一类是心跳同步，用于告诉 Relay：

- 当前用户在线
- 当前设备可连接
- 当前是否允许接待好友宠物
- 当前宠物是否可被派出
- 当前有哪些活跃串门会话

心跳同步不是同步完整宠物，只是同步在线状态和接待能力。

第二类是事件同步，只在发生事情时触发：

- 主人派宠物出门
- 好友宠物到达
- 用户点击、拖拽、投喂或请回宠物
- 来访宠物和本地宠物发生互动
- 串门结束
- 互动事件回传
- 本地发布新的 `pet_profile` 快照

因此，第一版不需要持续把宠物状态刷到服务器。服务器只需要知道谁在线、谁在访问谁、发生了哪些需要路由的事件。

### 6.8 第一版串门闭环

第一版串门可以先做成主人主动派出。

最小闭环：

1. 用户有一个好友列表。
2. 用户选择好友，点击让宠物去串门。
3. 本地 Runtime 向 Relay 创建 `visit_session`。
4. Relay 判断好友在线、好友关系和自动来访权限。
5. 好友 Runtime 收到来访请求，并创建来访宠物投影。
6. 好友在桌面上真实看到这只宠物。
7. 好友可以点击、拖拽、投喂或请回它。
8. 好友 Runtime 记录互动事件。
9. 串门结束后，事件通过 Relay 回传给主人 Runtime。
10. 主人 Runtime 将事件转成生活日志、记忆和宠物关系牵连。

### 6.9 待继续讨论的问题

以下问题还没有完全确定：

- 后续版本是否允许宠物主动决定出门？
- 串门是否需要停留时间上限？
- 目标用户离线时，真实串门如何处理？
- 来访宠物和本地宠物之间的互动，哪些由 Runtime 自动生成，哪些需要 Agent 参与解释？
- 互动事件是实时回传给主人，还是等宠物回家后统一回传？
- 好友关系如何建立？
- 用户是否可以给来访宠物投喂礼物？
- 礼物是纯本地记录，还是会成为宠物可携带的资源？
- 宠物是否可以同时访问多个桌面？

## 7. 生活日志

生活日志是宠物连续生活线的主要呈现方式。

它记录宠物今天真实发生过的事情：

- 在家做了什么
- 去了谁的桌面
- 遇见了哪只宠物
- 被谁互动了
- 收到了什么礼物
- 产生了什么记忆
- 回家后心情如何

生活日志应该有情绪价值，但不应该伪造真实社交事件。真实发生的来访互动，需要和纯粹由本地模拟生成的离线生活片段区分清楚。

## 8. 关系图谱

关系图谱只记录宠物之间的互动牵连，不评价人际关系。

它不应该说：

- 你和某某关系很好。
- 你们亲密度下降了。
- 你应该多联系某人。

它可以说：

- Momo 最近 7 天去了小柚那里 3 次。
- 小柚来访时，Momo 通常会靠近它。
- 它们一起停在屏幕左下角的次数变多了。
- Momo 收到过小柚带来的两枚星星贴纸。
- 它们有一个共同记忆：第一次在雨天主题桌面见面。

关系图谱可以记录：

- 见面次数
- 最近见面时间
- 共同停留位置
- 共同互动事件
- 礼物往来
- 共同记忆
- 宠物之间形成的习惯

原则：

> 关系图谱不做排名，不制造社交压力，只记录宠物之间生成出来的牵连。

## 9. 离线与休眠生活

宠物即使在电脑合盖、进程休眠或用户离开时，也应该拥有连续生活感。

可以有两类机制：

### 9.1 轻量模拟

当用户回来时，Runtime 根据离开时间生成一段合理生活片段：

- 离开 20 分钟，宠物睡了一觉。
- 离开 3 小时，宠物整理了自己的小窝。
- 离开一晚上，宠物做了一个梦。

### 9.2 真实后台事件

如果通信服务和本地后台进程允许，宠物可能在用户离开时真实完成某些事件：

- 收到好友宠物来访
- 去好友桌面串门
- 回传记忆

这部分需要后续根据实现成本继续讨论。

## 10. 下一步需要继续聊清楚的内容

优先继续讨论：

1. 宠物到底如何发起一次串门。
2. 好友关系和自动来访权限如何建立。
3. 来访宠物在宿主桌面上的行为如何生成和约束。
4. 互动事件如何变成宠物自己的记忆。
5. Pet Protocol 中“一只宠物”的概念定义。
6. Pet Skill 第一版需要生成到什么程度。
